Требуемые условия завершения

MVP-требования к собственной СУБД

1. Постоянное хранилище

Данные должны сохраняться между запусками системы. Формат хранения — на усмотрение разработчика (бинарный файл, JSON, CSV, собственный формат и т.д.). При перезапуске СУБД все ранее созданные таблицы и записи должны быть доступны.

2. C++ библиотека для взаимодействия с СУБД

Всё взаимодействие клиента с СУБД должно быть оформлено в виде отдельной C++ библиотеки (статической .a или динамической .so / .dylib). Клиентский код должен работать с СУБД исключительно через API этой библиотеки — без прямого обращения к файлам хранилища или внутренним структурам.

3. Паттерн проектирования

В архитектуре проекта должен быть явно применён хотя бы один паттерн проектирования (GoF или архитектурный). В отчете необходимо указать:  
- название паттерна;  
- где именно он применён;  
- почему выбран именно этот паттерн.

4. Скрипты сборки и запуска

Сборка проекта должна осуществляться посредством запуска build.sh. Скрипт должен:  
1. Определять операционную систему и доступный пакетный менеджер (apt, brew, pacman и т.п.);  
2. Устанавливать все необходимые системные зависимости (компилятор, CMake, сторонние библиотеки и т.д.);  
3. Загружать зависимости через менеджер пакетов C++ (vcpkg, Conan и т.п.) — при их использовании;  
4. Собирать проект.

Результатом работы скрипта должна быть папка build/, в которой будет находиться:  
1. Скомпилированная библиотека для работы с БД;  
2. Исполняемый файл СУБД.

Запуск системы осуществляется через скрипт (например, run.sh). Скрипт должен:  
1. Проверять на существование исполняемый файл СУБД, и если его нет, запускать build.sh.  
2. Принимать в себя параметры host и port, на которых будет запускаться СУБД;  
3. Запускать СУБД.

4. Поддерживаемые DDL-запросы

Грамматика (форма Бэкуса — Наура)

<ddl-statement>       ::= <create-database>  
                        | <drop-database>  
                        | <create-table>  
                        | <drop-table>

(* ───── DATABASE ───── *)  
<create-database>     ::= "CREATE" "DATABASE" <database-name> ";"

<drop-database>       ::= "DROP" "DATABASE" <database-name> ";"

(* ───── TABLE ───── *)  
<create-table>        ::= "CREATE" "TABLE" <table-name> "(" <column-def-list> ")" ";"

<column-def-list>     ::= <column-def>  
                        | <column-def-list> "," <column-def>

<column-def>          ::= <column-name> <data-type>

<data-type>           ::= "INT"  
                        | "FLOAT"  
                        | "BOOL"  
                        | "TEXT"  
                        | "VARCHAR" "(" <positive-integer> ")"

<drop-table>          ::= "DROP" "TABLE" <table-name> ";"

<database-name>       ::= <identifier>  
<table-name>          ::= <identifier>  
<column-name>         ::= <identifier>  
<identifier>          ::= <letter> { <letter> | <digit> | "_" }  
<letter>              ::= "A" | ... | "Z" | "a" | ... | "z"  
<digit>               ::= "0" | ... | "9"  
<positive-integer>    ::= <digit> { <digit> }

6. Поддерживаемые DML CRUD-запросы

Грамматика (форма Бэкуса — Наура)

<dml-statement>       ::= <select>  
                        | <insert>  
                        | <update>  
                        | <delete>

(* ───── SELECT ───── *)  
<select>              ::= "SELECT" <projection>  
                          "FROM" <table-name>  
                          [ "WHERE" <condition> ] ";"

<projection>          ::= "*"  
                        | <column-list>

<column-list>         ::= <column-name>  
                        | <column-list> "," <column-name>

(* ───── INSERT ───── *)  
<insert>              ::= "INSERT" "INTO" <table-name>  
                          [ "(" <column-list> ")" ]  
                          "VALUES" <values-row-list> ";"

<values-row-list>     ::= <values-row>  
                        | <values-row-list> "," <values-row>

<values-row>          ::= "(" <value-list> ")"

<value-list>          ::= <value>  
                        | <value-list> "," <value>

(* ───── UPDATE ───── *)  
<update>              ::= "UPDATE" <table-name>  
                          "SET" <assignment-list>  
                          [ "WHERE" <condition> ] ";"

<assignment-list>     ::= <assignment>  
                        | <assignment-list> "," <assignment>

<assignment>          ::= <column-name> "=" <value>

(* ───── DELETE ───── *)  
<delete>              ::= "DELETE" "FROM" <table-name>  
                          [ "WHERE" <condition> ] ";"

(* ───── Общие правила ───── *)  
<condition>           ::= <column-name> <comparison-op> <value>  
                        | <condition> "AND" <condition>  
                        | <condition> "OR" <condition>  
                        | "NOT" <condition>  
                        | "(" <condition> ")"

<comparison-op>       ::= "=" | "!=" | "<" | ">" | "<=" | ">="

<value>               ::= <integer-literal>  
                        | <float-literal>  
                        | <string-literal>  
                        | <bool-literal>  
                        | "NULL"

<integer-literal>     ::= [ "-" ] <positive-integer>  
<float-literal>       ::= [ "-" ] <positive-integer> "." <positive-integer>  
<string-literal>      ::= "'" { <any-char-except-quote> } "'"  
<bool-literal>        ::= "TRUE" | "FALSE"

7. Интерфейс пользователя

Интерфейс может быть консольным или графическим. Обязательные требования:

- Пользователь вводит запрос в текстовом виде.  
- Результат отображается в интерфейсе:  
  - для SELECT — табличное представление выбранных строк с заголовками столбцов;  
  - для INSERT / UPDATE / DELETE — сообщение с количеством затронутых строк, например: Affected rows: 3;  
  - для CREATE DATABASE / DROP DATABASE / CREATE TABLE / DROP TABLE — подтверждение успешного выполнения или сообщение об ошибке.

Чек-лист приёмки MVP

#  Критерий                                                Проверяется  
1   Данные сохраняются после перезапуска                   Перезапуск системы, проверка наличия данных  
2   Клиент использует C++ библиотеку                       Code review: нет прямого доступа к хранилищу вне библиотеки  
3   Применён паттерн проектирования                        Code review + описание в отчете  
4   build.sh устанавливает зависимости и собирает проект   Запуск на чистой машине без ручной установки пакетов  
5   Запуск через скрипт                                    ./run.sh поднимает систему без дополнительных действий  
6   CREATE DATABASE / DROP DATABASE работают               Функциональный тест  
7   CREATE TABLE / DROP TABLE работают                     Функциональный тест  
8   SELECT с проекцией и WHERE работает                    Функциональный тест  
9   INSERT с несколькими строками работает                 Функциональный тест  
10  UPDATE с WHERE работает                                Функциональный тест  
11  DELETE с WHERE работает                                Функциональный тест  
12  Интерфейс показывает таблицу / количество строк        Визуальная проверка
